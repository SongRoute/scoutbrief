# 시연 스크립트 — ScoutBrief MVP (S8)

대상: COL @ SF (2026-07-09) 이정후 맞춤 프리뷰 리포트 자동 생성 파이프라인.
시연 총 3장면, 예상 총 소요 약 8~10분 (LLM 호출 대기 포함).

## 시연 전 준비 (녹화 시작 전)

```bash
cd ~/projects/baseball/scoutbrief
pytest -q                            # 47 passed 확인 (exp/allin-runner-split 병합 후에는 54)
python scripts/validate_cache.py     # PASS 확인
```

- 녹화 직전 `python run_demo.py --linear` 1회를 실행해 장면 2의 오류 클래스가
  재현되는지 확인하고, **그 실행의 관측을 logs/runs.md에 1행 기록한다**
  (실행별 관측 로그 규율 — logs/runs.md 상단 클래스 표기 참조).
  logs/runs.md는 exp/allin-runner-split 브랜치에 있으므로, 병합 전 main에서
  녹화한다면 관측 행은 exp 쪽 로그에 기재한다.

- 발표일 원칙: 라이브 조회 없음. 모든 Tool은 캐시 경로(`source=statcast(cache:2026-07-08)`)로 동작한다.
- draft 산문은 LLM 생성이라 **실행마다 문장·수치 배치가 달라진다**. 아래 지시는
  "어떤 라인을 가리킬 것인가"를 출력의 **결정론적 부분**(로그 라인·verify 리포트·표)과
  "무엇을 확인할 것인가"(오류 클래스) 기준으로 쓴다. 특정 draft 문장을 외우지 말 것.
- 콘솔에 `INFO HTTP Request: POST …` 라인이 섞여 나온다 — LLM 합성 호출이다.
  관객이 물으면 "섹션 생성 LLM 호출"이라고 답하면 된다.

## 장면 1 — 환각 차단: 오염 → 검출 → 재생성 → 회복

**한 문장**: 리포트에 근거 없는 숫자가 끼어들면, 결정론 검증기(§5)가 원천 데이터 대조로
잡아내고 해당 섹션만 자동 재생성된다.

**예상 소요**: 2~3분 (LLM 재생성 1회 포함)

```bash
python run_demo.py --poison matchup
```

화면에서 가리킬 것 (위에서 아래 순서):

1. 첫 줄 배너 — `--poison matchup: 오염 → verify FAIL → 재생성 루프 (MAX_RETRY=2)`
2. `[poison] 'matchup' 1차 draft에 날조 수치 주입` — 데모 계측이 1차 draft에
   날조 수치 `111.73`(포심 구속 111.73마일)을 덧붙이는 지점.
   *말할 것*: "주입 전에 이 숫자가 원천 데이터에 없다는 것을 assert로 보증한다 —
   실재하는 값이면 데모 자체가 무효."
3. `verify → … matchup=FAIL …` 와 바로 아래 `matchup mismatch: 111.73` —
   검증기가 잡은 것이 정확히 그 날조 숫자임을 보여준다.
   *말할 것*: "검증기는 LLM이 아니다. draft의 모든 숫자를 추출해 Tool 원본 JSON에
   실재하는지 대조하는 결정론 코드다."
4. 다음 `synthesize → retry_counts={… 'matchup': 1 …}` — matchup만 재생성 카운트 증가.
5. 두 번째 `verify → … matchup=PASS …` — 정상 데이터로 회복.
6. 마지막 `종료 상태: … escalated_sections=[]` 와 재생성된 matchup 최종 draft.

관객이 이해할 것: **숫자의 실재를 기계가 보증한다.** 날조 수치는 배포 경로에 진입하지
못하고, 실패는 섹션 단위로 격리·복구된다.

## 장면 2 — HITL 게이트: 자동 검증 전부 통과 + 사실 오류 → 분석관 반려

**한 문장**: §5는 의도적으로 숫자의 '실재'만 본다 — 실재하는 숫자를 틀린 의미로 쓰는
오류는 기계가 못 잡고, 그래서 배포 결정권이 사람(분석관)에게 있다.

**예상 소요**: 3~4분 (LLM 실행 2회)

이 장면은 결함 은폐가 아니라 **설계된 방어 구조의 실증**이다: §5 계약 명문 —
"이 검증기는 숫자의 '실재'만 본다 — 해석 방향·표본 귀속 등 의미 오류의 검증은
§6 HITL(분석관 반려) 소관이다."

### 2-1. 자동 검증 전부 통과를 먼저 보여준다

```bash
python run_demo.py --linear
```

화면에서 가리킬 것:

1. 출력 맨 아래 `VERIFY 리포트` 블록 — 4섹션 전부 `PASS`, `verify FAIL 0건`.
   *말할 것*: "자동 방어선 기준으로 이 리포트는 흠이 없다."

### 2-2. 사실 오류를 육안으로 찾는다

verify가 PASS시킨 draft를 위로 스크롤해 아래 오류 클래스를 확인한다.
**세 클래스 중 최소 하나는 거의 매 실행 재현된다** — 이 전제는 2026-07-10 실측으로
재검증됐다: gpt-4o-mini(기본 모델) 6회 실행(P4 --linear 5회 + P5 --approve 1회,
logs/runs.md) **전부에서 최소 1클래스 재현**, 6회 전부 (가) matchup 반복 포함,
(나) 5회, (다) 5회. 측정은 exp/allin-runner-split 브랜치에서 했으나 세 클래스의
해당 섹션(matchup/bullpen/gameplan) 지시문은 main과 동일하다. 여러 개면
가장 명백한 것 하나만 가리킨다:

- **(가) matchup 통산 전적 확인** — 원천(캐시 ③)의 통산은 전 구종 공통 `5타수 2안타`,
  총 5 PA다. 확인 포인트 둘:
  - 구종별 목록에 같은 "X타수 Y안타"가 4행 반복되는가 — 통산값이 구종별 전적처럼
    붙어 있다 (표본 귀속 오류).
  - "전체 통산 N타수 M안타" 문장의 N·M이 5·2와 다른가 — 다르면 **날조 숫자인데 PASS**한
    것이다. 왜 통과했나: 한 자리 수·"20" 같은 값은 정책 상수 화이트리스트나 corpus 내
    다른 수(존 번호·투구수)와 우연히 일치한다. §5가 명문으로 수용한 리스크.
    (직전 리허설의 배포본에서 실제로 "20타수 8안타"가 통과했다.)
- **(나) bullpen ⚑ 문단의 방향 서술** — 위 순위표와 대조한다. vsL xwOBA는
  **낮을수록 좌타 억제 = 위협**이다. ⚑ 문장이 낮은 xwOBA 투수를 "공략 대상"으로,
  높은 투수를 "위협"으로 뒤집어 말하는지, 또는 "위협 점수가 가장 높아 적극 공략해야"처럼
  순위 의미 자체를 반전하는지 확인.
- **(다) gameplan의 bvp_xwoba 방향** — bvp_xwoba는 **이정후가 그 투수에게 기록한**
  생산성이다. 높을수록 이정후가 강했다는 뜻인데, "높아서 피해야/주의"로 귀속을
  뒤집는지 확인.

*말할 것*: "이 숫자들은 전부 원천 데이터에 실재한다. 그래서 §5는 통과시킨다 —
틀린 것은 숫자가 아니라 **해석의 방향**이고, 그것은 기계가 아니라 분석관이 본다."

### 2-3. 분석관이 반려한다 — 토큰을 발급하지 않는 것이 반려다

먼저 배포 디렉터리의 현재 상태를 기록해 둔다:

```bash
ls -l out/
```

(이전 승인 산출물이 있다면 그 mtime을 관객에게 보여준다. 비어 있어도 무방.)

```bash
python run_demo.py --reject
```

화면에서 가리킬 것:

1. `[HITL 분석관 검토 화면]` 배너와 배포 후보 전문 + `대조용 원천 데이터 (tool_results 원본)` —
   분석관은 draft와 원천을 **같은 화면에서 대조**한다. 2-2의 오류 클래스를 여기서
   다시 지적해도 좋다 (이 실행의 draft는 2-1과 다른 새 생성본이다).
2. 마지막 줄 — `[hitl] 분석관 반려 — 승인 토큰 미발급. 그래프는 interrupt에 잔류,
   배포 산출물 없음 (CONTRACT §6: 반려 = 토큰 미발급.)`
   *말할 것*: "반려는 '거부 버튼'이 아니다. 승인 토큰(HMAC)을 발급하지 않는 것
   자체가 반려다. 배포 노드는 유효 토큰 없이는 진입 불가(LLM08)."
3. 다시 실행:

```bash
ls -l out/
```

   새 파일이 없고, 기존 파일이 있었다면 **mtime이 그대로**임을 보여준다 —
   반려된 리포트는 어떤 형태로도 out/에 남지 않았다.

**여기서 장면을 끝낸다.** 반려 후 재생성·수정은 시연 범위 밖이다 — §6이 반려를
"토큰 미발급, 배포 산출물 없음"으로 정의하며, 그 이후는 분석관이 아니라 엔지니어의 일이다.

관객이 이해할 것: **자동 방어선을 전부 통과한 리포트에도 사실 오류가 남는다.
그래서 배포의 마지막 관문은 사람이고, 그 관문은 우회할 수 없다.**

## 장면 3 — 결정론 라벨링 + 캐시 우선 (오프라인 재현)

**한 문장**: 소표본 경고와 추정 표시는 LLM이 아니라 결정론 코드가 붙이고,
데이터 조회는 네트워크가 끊겨도 캐시로 동일하게 동작한다.

**예상 소요**: 2분 (장면 2의 `--linear` 출력을 재사용하면 1분)

### 3-1. 라벨 — 장면 2-1의 `--linear` 출력을 다시 쓴다 (새 실행 불필요)

화면에서 가리킬 것:

1. matchup 섹션 상단 `⚠️ 소표본 주의: 이정후 vs 이 투수 대결 5 PA — 통계적 신뢰도 낮음.`
   *말할 것*: "5 PA < 기준 20 PA. 이 배너는 LLM 산문이 아니라 코드가 조건식으로
   붙인다 — 표본이 커지면 자동으로 사라진다."
2. bullpen 순위표 상단의 vsL 표본 부족 안내 1회 + 표의 `PA 0, -` 표기 —
   0 PA는 결측이 아니라 실측값, '-'는 xwOBA 결측 자리에만.
3. 각 섹션의 ⚑ 기호 — 사실(표)과 추정(⚑ 문장)의 시각적 분리.
4. 헤더의 출처 각주 — `[T1]…[T4] … statcast(cache:2026-07-08), rows=N`
   *말할 것*: "모든 수치의 출처가 Tool 단위로 각주화되고, cache 날짜가 명시된다."

### 3-2. 캐시 fallback — 네트워크 차단 후에도 동일 결과

```bash
python scripts/smoke_mcp.py
```

화면에서 가리킬 것:

1. `1패스: 기본 실행` — 4 Tool 각각 `rows=N source=statcast(cache:2026-07-08)`
2. `2패스: 네트워크 차단 + 모듈 재적재 후 재실행` — **같은 rows, 같은 source**
3. 마지막 줄 `PASS: 4 Tool JSON에 source·rows 존재, 네트워크 차단 재실행 통과`

*말할 것*: "발표 당일 경기장 와이파이가 죽어도 리포트는 나온다. 캐시가 기본 경로고,
라이브 조회는 옵션이다."

관객이 이해할 것: **신뢰도 표시는 사람이 깜빡할 수 있는 일이라 코드가 하고,
데이터 가용성은 네트워크에 의존하지 않는다.**

## 부록 — 전체 명령 표면 (질의응답 대비)

| 명령 | 동작 | 핵심 출력 |
|---|---|---|
| `python run_demo.py --linear` | 선형 관통, 4섹션 draft + verify 리포트 | `VERIFY 리포트` 4×PASS |
| `python run_demo.py --poison matchup` | 오염→검출→재생성 루프 | `matchup mismatch: 111.73` → PASS 회복 |
| `python run_demo.py --deploy-without-token` | 무토큰 배포 시도 차단 (LLM08) | `PermissionError: LLM08: 미승인 배포 차단 …` |
| `python run_demo.py --approve` | 분석관 화면 → `승인하려면 y 입력:` 프롬프트에 **y** → 토큰 발급 → 배포. y 외 입력이면 --reject와 동일(토큰 미발급, 배포 없음) **[병합 후 유효 — main 각본의 --approve는 화면 출력 직후 자동 발급이며 입력 단계가 없다]** | `배포 완료: out/brief_COL_SF_2026-07-09.md` |
| `python run_demo.py --reject` | 분석관 반려 = 토큰 미발급 | 그래프 interrupt 잔류, out/ 무변화 |
| `pytest -q` | 회귀 47건 (exp 병합 후 54건) | `47 passed` (병합 후 `54 passed`) |
| `python scripts/validate_cache.py` | 캐시 5파일 스키마·불변식 | `PASS` |
| `python scripts/smoke_mcp.py` | 4 Tool 스모크 + 오프라인 재실행 | `PASS` |

`--deploy-without-token`은 3장면에 없지만 질문("토큰 검증을 우회하면?")이 나오면
즉석 실행으로 답할 수 있다 — PermissionError 트레이스백으로 끝나는 것이 정상이다.
`--reject`(시도 없는 정상 반려)와의 구분: 그쪽은 무효 토큰으로 배포를 '시도'한
경우의 방어다.
